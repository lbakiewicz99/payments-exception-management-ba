# Business Rules

## 1. Purpose

This document defines the business rules for the payment exception management workflow.

Business rules describe how payment exception cases should be categorized, prioritized, assigned, monitored, escalated, and closed.

There rules support consistent operational handling, SLA monitoring, management reporting, and control evidence.

## 2. Business rule categories

| Category               | Description                                                          |
|------------------------|----------------------------------------------------------------------|
| Case creation rules    | Rules for creating and identifying payment exception cases           |
| Status lifecycle rules | Rules for allowed case status transitions                            |
| Priority rules         | Rules for assigning Low, Medium, High, or Critical priority          |
| SLA rules              | Rules for calculating SLA due dates and breach flags                 |
| Aging rules            | Rules for classifying long-running open exceptions                   |
| Ownership rules        | Rules for assigning and monitoring case ownership                    |
| Escalation rules       | Rules for identifying cases requiring team lead or manager attention |
| CLosure rules          | Rules for resolving investigation and closing cases                  |
| Audit trail rules      | Rules for retaining investigation and status history                 |
| Reporting rules        | Rules supporting operational and management reporting                |

## 4. Status lifecycle rules

Allowed statuses:

| Status                    | Description                                                                                    |
|---------------------------|------------------------------------------------------------------------------------------------|
| New                       | Case has been created but investigation has not started                                        |
| In Review                 | Analyst is actively investigating the case                                                     |
| Pending External Response | Case is waiting for input from bank, PSP, client, or another party                             |
| Escalated                 | Case requires attention from team lead, manager, risk, compliance, or another escalation owner |
| Resolved                  | Investigation outcome is known, but the case is not formally closed                            |
| Closed                    | Case is completed with required notes and closure reason                                       |

Status transition rules"

| Rule ID | Rule                                                                                                                         | Related requirements |
|---------|------------------------------------------------------------------------------------------------------------------------------|----------------------|
| BR-007  | A new case must start with status New.                                                                                       | FR-001, FR-005       |
| BR-008  | A case may move from New to In Review once minimum required fields are completed.                                            | FR-005               |
| BR-009  | A case may move from In Review to Pending External Response if external input is required.                                   | FR-005               |
| BR-010  | A case may move from Pending External Response back to In Review once the required response is received.                     | FR-005               |
| BR-011  | A case may move from In Review or Pending External Response to Escalated when escalation criteria are met.                   | FR-012               |
| BR-012  | A case may move from Escalated back to In Review when escalation review is completed and further analyst action is required. | FR-005               |
| BR-013  | A case may move from In Review or Escalated to Resolved once the investigation outcome is confirmed.                         | FR-005               |
| BR-014  | A case may move from Resolved to Closed only when closure requirements are met.                                              | FR-013               | 
| BR-015  | A case must not move directly from New to Closed.                                                                            | FR-014               |
| BR-016  | A closed case must not be edited except through a controlled reopen process.                                                 | NFR-003, CR-007      |

Allowed status transitions:

| From status               | Allowed next status                            |
|---------------------------|------------------------------------------------|
| New                       | In Review                                      |
| In Review                 | Pending External Response, Escalated, Resolved |
| Pending External Response | In Review, Escalated                           |
| Escalated                 | In Review, Resolved                            |
| Resolved                  | Closed                                         |
| Closed                    | No standard transition                         |

## 5. Priority rules

Priority values:

| Priority | Description                                                         |
|----------|---------------------------------------------------------------------|
| Low      | Low operational impact, no urgent action required                   |
| Medium   | Standard investigation priority                                     |
| High     | Time-sensitive or material issue requiring prompt review            |
| Critical | Severe business, financial, customer, regulatory, or control impact |

Priority assignment rules:

| Rule ID | Rule                                                                                                                                         | Related requirements |
|---------|----------------------------------------------------------------------------------------------------------------------------------------------|----------------------|
| BR-017  | Every payment exception case must have a priority.                                                                                           | FR-007, DR-010       |
| BR-018  | Priority should be assigned based on amount, exception type, payment channel, customer impact, SLA sensitivity, and operational risk.        | FR-007               |
| BR-019  | Cases with payment amount greater than or equal to 100,000 EUR equivalent must be assigned at least High priority.                           | CR-003               |
| BR-020  | Cases with confirmed duplcate payment risk must be assigned at least High priority.                                                          | CR-003               |
| BR-021  | Cases with potential regulatory, compliance, or client-impacting risk must be assigned Critical priority.                                    | CR-003               |
| BR-022  | Cases impacting settlement, cash availability, or customer payment completion may be assigned High or Critical priority depending on impact. | CR-003               |
| BR-023  | Manual priority changes must be recorded in case history.                                                                                    | CR-008               |
| BR-024  | Priority cannoit be blank for open cases.                                                                                                    | DR-010               |

Suggested priority matrix:

| Condition                                               | Suggested priority |
|---------------------------------------------------------|--------------------|
| Low amount, no customer impact, no SLA risk             | Low                |
| Standard payment issue requiring investigation          | Medium             |
| High-value payment exception                            | High               |
| Duplicate payment risk                                  | High               |
| Aged case with unresolved investigation                 | High               |
| Regulatory, compliance, fraud, or major customer impact | Critical           |
| Payment issue affecting settlement or cash availability | High/Critical      |

## 6. SLA rules

SLA rules define how quickly cases should be reviewed or resolved.

SLA targets:

| Priority | Initial review SLA | Resolution target |
|----------|--------------------|-------------------|
| Critical | Same business day  | 1 business day    |
| High     | 1 business day     | 3 business days   |
| Medium   | 2 business days    | 5 business days   |
| Low      | 3 business days    | 10 business days  |

SLA business rules:

| Rule ID | Rule                                                                                                    | Related requirements   |
|---------|---------------------------------------------------------------------------------------------------------|------------------------|
| BR-025  | SLA due date must be calculated from the case created date and priority.                                | FR-008, DR-013         |
| BR-026  | SLA calculation must use business days, excluding weekends.                                             | FR-008                 |
| BR-027  | A case is SLA breached when the current date is later than the SLA due date and the case is not Closed. | FR-009, DR-014, CR-004 |
| BR-028  | SLA breach flag must be visible in the exception queue.                                                 | FR-015, RR-004         |
| BR-029  | SLA-breached case must be included in operational and management reporting.                             | RR-004                 |
| BR-030  | Critical and High priority SLA breaches must be escalated.                                              | FR-012, CR-003, CR-004 |
| BR-031  | Changing priority must recalculate the SLA due date unless the case is already closed.                  | FR-008                 |
| BR-032  | SLA due date must not be blank for open cases.                                                          | DR-013                 |

## 7. Aging rules 

Aging rules classify open cases based on how long they have remained unresolved.

Aging bucket definitions:

| Aging bucket | Definition                                        |
|--------------|---------------------------------------------------|
| 0-2 days     | Case has been open for 0 to 2 business days       |
| 3-5 days     | Case has been open for 3 to 5 business days       |
| 6-10 days    | Case has been open for 6 to 10 business days      |
| 11-30 days   | Case has been open for 11 to 30 business days     |
| Over 30 days | Case has been open for more than 30 business days |

Aging business rules:

| Rule ID | Rule                                                                            | Related requirements   |
|---------|---------------------------------------------------------------------------------|------------------------|
| BR-033  | Case age must be calculated from created date to current date for open cases.   | DR-015                 |
| BR-034  | Closed cases should retain final age at closure for reporting.                  | DR-019                 |
| BR-035  | Cases open for more than 5 business days must be classified as aged exceptions. | FR-010, CR-005         |
| BR-036  | Aged cases must be visible in reporting.                                        | RR-003                 |
| BR-037  | Aged High or Critical priority cases must be escalated.                         | FR-012, CR-003, CR-005 |
| BR-038  | Aging bucket must be recalculated daily for open cases.                         | DR-015                 |

| Required item       | Description                                   |
|---------------------|-----------------------------------------------|

| Investigation notes | Notes must describe the investiation outcome  |
| Closure reason      | Controlled closure reason must be selected    |

| Owner               | Case must have an owner or closing user       |

Closure business rules:

| Rule ID | Rule                                                               | Related requirements   |
|---------|--------------------------------------------------------------------|------------------------|
| BR-052  | A case can only be closed from status Resolved.                    | FR-013, FR-014         |

| BR-054  | A case cannot be closed without investigation notes.               | DR-017, CR-002         |
| BR-055  | Closed date must be recorded when a case is closed.                | DR-019                 |
| BR-056  | Closure reason must be selected from a controlled list.            | DR-018                 |

Suggested closure reasons:

| Closure reason          | Description                                             |
|-------------------------|---------------------------------------------------------|
| Payment settled         | Payment was completed or confirmed as settled           |
| Payment reprocessed     | Payment was corrected and reprocessed                   |
| Payment cancelled       | Payment was cancelled and no further action is required |

| Bank rejection resolved | Bank rejection reason was addressed                     |
| PSP issue resolved      | PSP-related issue was resolved                          |

## 13. Exception taxonomy rules

Suggested exception types:

| Exception type         | Description                                                          | 
|------------------------|----------------------------------------------------------------------| 
| Bank Rejection         | Payment rejected by bank                                             |
| PSP Rejection          | Payment rejected or failed at Payment Service Provider level         |
| Returned Payment       | Payment returned after initiation                                    |
| Failed Payment         | Payment failed before completion                                     |
| Delayed Settlement     | Payment or settlement confirmation delayed                           |
| Duplicate Payment Risk | Potential duplicate payment identified                               |
| Unmatched Payment      | Payment cannot be matched to expected internal or external record    | 
| Incorrect Amount       | Amount mismatch identified                                           |
| Incorrect Currency     | Currency mismatch identified                                         |
| Missing Reference      | Required payment reference is missing                                |
| Customer Query         | Exception raised through customer support or client query            |
| Manual Review Required | Case requires manual operations review but does not fit another type |

Taxonomy rules:

| Rule ID | Rule                                                                     | Related requirements |
|---------|--------------------------------------------------------------------------|----------------------|
| BR-075  | Exception type must be selected from a controlled list.                  | FR-004, DR-008       |
| BR-076  | Exception type must not be free text in the standard workflow.           | FR-004               |
| BR-077  | New exception types must be reviewed before being added to the taxonomy. | NFR-006              |
| BR-078  | Exception type must be available for reporting.                          | RR-005               |

## 15. Summary 

These business rules provide the operational logic required to support a controlled payment exception management workflow.

They define how cases are created, categorized, prioritized, assigned, monitored, escalated, reported, and closed.

The rules also support traceability between business objectives, functional requirements, controls, reporting, and UAT test cases.
